## 배경
일반 코딩 에이전트의 하네스 엔지니어링 (지시 문서·아키텍처 제약·피드백 루프·지식 저장소·가비지 컬렉션) 은 *정적 환경 설계* 다. 환경은 *사람이* 유지·갱신해야 한다.

JARVIS 는 *자기 자신을 운영하는 자율 멀티 에이전트* 라서:
- 사람이 매번 헌법을 갱신·검증·정제하기엔 *작업량이 누적적*
- 운영 회차마다 *어떤 패턴이 자주 반복되는지* 사람이 눈으로 추적 불가
- 헌법 정합성이 *시간 지날수록 떨어지는* 자연적 드리프트 발생

*하네스가 스스로 진화하는 구조* 가 필요했다.

## 결정
JARVIS07 GUARDIAN 을 *Self-Evolving Harness* 의 단일 진입점으로 박제. 환경 자체가 *학습·정제·진화* 하는 폐쇄 루프 구축.

### 비전
> 시간이 지날수록 *스스로 똑똑해지는* 에이전트. 매 회차 진단·수정 결과가 *영구 자산* 으로 누적되어 다음 회차에 활용되는 *폐쇄 학습 루프* 구축.

### 진화 루프
```
[오류 발생] → [error_collector] → [pattern_fixer Tier 1·2·3]
                                      ↓
                            [fix 성공] → [learned_patterns 자동 등록]
                                              ↓
                          [다음 회차 자가 진단 시 Layer 5 점검]
                                              ↓
                  [노이즈 정리 / 5회+ 반복 → 새 _fix_*() 자동 신설]
                                              ↓
                          [LLM 호출 0 — 즉시 fix 가능 영역 확장]
                                              ↓
                          [self_repair_runs 메트릭 누적]
                                              ↓
                          [hub.py 대시보드 학습 곡선 표시]
                                              ↓
                          [사용자가 학습 효과 *눈으로* 확인]
```

### 7-Layer 자가 진단 (auto_repair.py 매일 08:30 / 18:00)
1. Syntax & Import 정합성
2. CLAUDE.md 규정 위반 (27종 grep 검증)
3. 분량 표기 표준 (제8-B조)
4. 최근 발행 글 품질 회귀 (post_analysis DB 검사)
5. learned_patterns 데이터 정합성 (노이즈 검출)
6. 메타 학습 기회 (5회+ 반복 → 새 fixer 자동 제안)
7. 비전 정합성 (3회+ 반복 교훈 → 헌법 자동 박제 + 검증 명령 추가)

## 이유
1. **운영 누적 부담 0**: 사람이 매번 헌법·검증 갱신할 필요 없음. GUARDIAN 이 자동 수행.
2. **표준 하네스 → 진화형**: 정적 환경 → *환경이 스스로 자기 진단* + 자기 정제. 표준 위 단계.
3. **메트릭 시각화**: hub.py 학습 곡선 카드로 *학습 효과 정량* 표시. "정말 똑똑해지고 있는가" 가 *눈으로 확인 가능*.
4. **사용자 신뢰 누적**: 매 회차 *어떤 학습이 일어났는지* 보고. 신뢰가 시간 축적으로 누적.

## 포기한 대안
1. **사람 주도 헌법 갱신**: 작업량 누적. 자율성 의미 약화. 포기.
2. **단일 모델 자가 진단** (Haiku 만): 메타 학습·헌법 정제는 추론 깊이 필요 — Opus 4.6 필수. 포기.
3. **외부 분석 도구 (SonarQube 등)**: JARVIS 의 자체 학습 자산화와 통합 어려움. 진화 루프 닫기 불가. 포기.
