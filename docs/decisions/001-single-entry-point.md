# 001. 단일 진입점 원칙 — 모든 도메인 책임을 하나의 폴더로

## 상태
확정 (2026-04 ~ 2026-05 박제, 헌법 7회 강화)

## 배경
JARVIS 가 다중 에이전트로 확장되면서 *같은 책임* (스케줄·인프라·이미지·분량·도구 등) 이 *여러 폴더에 분산* 되는 사고가 반복됐다. 분산된 책임은:
- 이중 capability 등록 (인프라 capability 두 곳에서 declare → 데몬 부팅 시 충돌)
- EventListener 중복 부착 (job_runs DB 중복 적재)
- 분량 정책 위반 잔존 (length_manager 외에서 한국어 글자 자르기)
- 이미지 폴백 체인 미작동 (다른 파일에서 Pollinations 직접 호출)
같은 형태로 *반복적인 운영 사고* 를 일으켰다.

3회 이상 동일 유형 사고가 박제된 직후 (헌법 박제 2026-05-15) 사용자 정책으로 *단일 진입점 원칙* 이 절대 헌법으로 격상됐다.

## 결정
시스템 내 *모든 도메인 책임* 은 *지정된 단 하나의 폴더·파일* 만 보유한다. 다른 위치에서 같은 책임을 코드로 구현·재정의·우회하는 행위 일체 금지.

도메인 → 단일 진입점 매핑:

| 도메인 | 단일 진입점 |
|--------|------------|
| 인프라 (데몬·프로세스·시스템 상태) | `JARVIS00_INFRA/infra_agent.py` |
| 스케줄 (cron·interval·polling) | `JARVIS04_SCHEDULER/` |
| 이미지 생성 (AI 사진·차트·썸네일) | `JARVIS06_IMAGE/image_agent.py` |
| 분량 (글자수·문장수·압축) | `JARVIS02_WRITER/length_manager.py` |
| 블로그 정책 (14조 헌법) | `JARVIS02_WRITER/BLOG_SUPREME_LAW.md` |
| 오류·자가수정 | `JARVIS07_GUARDIAN/` |
| 도구 등록 (라우터용) | `shared/tools.py` + `JARVIS01_MASTER/agent_tools.py` |
| 메모리 시스템 | 사용자 메모리 디렉토리 |

## 이유
1. **재발 방지 구조**: 분산 → 다음 작업자가 *어디서 구현해야 할지* 헷갈림 → 임시 위치에 박음 → 사고 재발. 단일 진입점은 *물리적으로 분산할 곳이 없다.*
2. **검증 자동화 가능**: `grep -v <단일 진입점 폴더>` 만으로 위반 잔존 즉시 검출. `shared/precommit_check.py` 의 27종 검증이 이 원칙 위에 구축됨.
3. **이관 의무 명확**: 분산된 코드 발견 시 *즉시 단일 진입점으로 이관* — 미루는 행위 자체가 규정 위반. 위치 결정이 모호하지 않다.
4. **사용자 정신적 부담 감소**: "이 책임은 어디 있지?" 를 *물어볼 필요 없음*. 도메인 → 폴더 매핑이 항상 1:1.

## 포기한 대안
1. **레이어드 책임 분산**: 책임을 *공통 모듈* + *각 에이전트 어댑터* 두 곳에 박는 방식. 어댑터 변경이 누적되면 결국 분산. 포기.
2. **인터페이스 추상화**: 추상 인터페이스 + 다중 구현. 분산이 의도된 경우 (예: image provider 3개) 만 정당. 도메인 자체에는 *과잉 설계* — 포기.
3. **공유 라이브러리 (shared/)에 모두**: shared 가 너무 비대해지고, 도메인 경계가 흐려짐. shared 는 *도메인 무관 유틸* 만 보유. 포기.
