# 006. 사용자 박제 4원칙

## 상태
확정 (2026-05-15 사용자 직접 박제, 메모리 `feedback_user_pinning_principles.md`)

## 배경
JARVIS 의 *헌법 (CLAUDE.md)* 과 *정책 (BLOG_SUPREME_LAW.md)* 이 누적되면서, 박제 행위 자체에 *방법론적 일관성* 이 필요해졌다. 잘못된 박제는:
- *금지 사항 나열* — 새 우회 경로가 발견될 때마다 *금지 항목 추가* → 박제가 비대해지면서 핵심 의도 희석
- *단일 진입점 박제 후 실제 위반 잔존* — 박제만 하고 실제 코드 점검 안 함 → 헌법 위반 누적
- *결정적 seed 박제* — 시간·날짜 등 외부 변동 가능 값에 의존하는 박제 → 박제 시점 이후 변경 시 정합성 깨짐
- *외부 인증 빈발 박제* — 사용자에게 매번 확인 요구 → 자율성 의미 상실

3회 이상 반복된 박제 사고 직후 사용자가 직접 박제 4원칙을 제시했다.

## 결정
모든 박제 (CLAUDE.md / BLOG_SUPREME_LAW / docs/decisions / 메모리) 는 다음 4원칙을 따른다.

### 원칙 1 — 허용 패턴 유일 + 금지 다수 박제
*허용되는 패턴은 단 하나만 명시*. 그 외 모든 시도는 *자동 금지*. 새 우회 경로 발견 시 금지 항목 추가가 아닌 *허용 패턴 외 = 모두 금지* 가 기본값.

> 잘못된 박제: "FALLBACK_TEXT 금지, FALLBACK_HTML 금지, _CTA_POOL 금지 ..."  
> 올바른 박제: "허용 = `shared.llm.invoke_text("writer_fast", ...)` + 1줄 비상 폴백만. 그 외 모든 고정 풀·고정 템플릿 금지."

### 원칙 2 — 단일 진입점 박제 후 *실제* 확인
박제 본문만 추가하지 말고, *반드시* 검증 명령 (grep) 실행해서 *현재 잔존 위반 0건* 확인. 박제와 코드 정합성이 *동시에* 맞춰져야 박제 완료.

> 박제 직후 의무: `grep -rnE '<위반 패턴>' --include='*.py' .` 결과 0행 검증.  
> 잔존 시: *박제 완료 아님*. 잔존 코드 이관 후 다시 검증.

### 원칙 3 — 결정적 seed 박제 금지
박제 본문에 *시간·날짜·환경별 변동 값* 박는 행위 금지. 시간이 지나면 박제 시점 정보가 stale 됨.

> 잘못된 박제: "데몬은 매일 08:30·18:00 자동 진단 (오늘 06:30 부팅 기준)"  
> 올바른 박제: "데몬은 매일 08:30·18:00 자동 진단" + 부팅 시점은 *동적 조회*.

### 원칙 4 — 외부 인증 최소 빈도
사용자에게 *매번* 확인을 요구하지 말고, *원칙 1~3 을 박제* 한 후 *자비스가 자율 판단*. 인증 빈도는 *외부 영향 도구* 의 텔레그램 ✅/❌ 게이트 외 최소화.

> 잘못된 패턴: "이 작업 진행해도 되나요?" 매 단계 확인  
> 올바른 패턴: 사용자 지시 1회 → 자비스가 박제 본문 따라 자율 진행 → 외부 영향 도구만 ✅ 게이트.

## 이유
1. **헌법 비대화 차단**: 금지 다수 박제는 *예외 카탈로그* 화 → 시간이 지나면 어떤 게 핵심인지 불명. 허용 유일 박제는 *영구 명확*.
2. **박제·코드 정합성 보장**: "박제 후 실제 확인" 의무는 *박제가 단순 선언이 아닌 *적용 의무* 임을 명시.
3. **시간 독립성**: 결정적 seed 박제 금지는 *박제 본문이 시간 지나도 유효* 하게 유지.
4. **자율성 의미 보존**: 외부 인증 최소 빈도는 *자비스의 존재 이유* 자체. 매번 묻는 자비스는 단순 매크로.

## 포기한 대안
1. **박제 형식 표준화만 (4원칙 없이)**: 형식만 통일해도 *내용의 일관성* 부족. 포기.
2. **자동 박제 (자비스가 새 패턴 발견 시 자동 박제)**: 박제는 *사용자 결정* — 자비스 자율 영역 밖. 포기.
3. **박제 만료 (N개월 후 자동 재검토)**: 박제는 *의도된 영구 결정*. 만료는 결정의 의미 약화. 폐기·대체는 명시적 ADR 로만. 포기.
