## 배경
블로그 글 작성·발행 정책은 *수십 개의 미세 규칙* (감성 도입부·문단 구조·교차 배치·여백·분량 등) 이 모여 *한 편의 글 품질* 을 결정한다. 이 정책들이:
- CLAUDE.md 안에 분산 → 정책 본문이 비대해져 *코드 규정* 과 *글 정책* 이 섞임
- 각 작성자 (jarvis_main / trend_economic_writer / tistory_html_writer) 안에 박힘 → 작성자별 미세 일관성 차이
- prompt 작성 시 매번 자연어로 박음 → 정책 1줄 변경에도 모든 prompt 수정 필요

정책의 *물리적 단일 진실 소스* 부재가 *품질 회귀의 근원* 이었다.

## 이유
1. **단일 진실 소스**: 정책 변경 시 *14조 본문 1줄만 수정* → 다음 발행부터 자동 적용. 작성자 코드 수정 불필요.
2. **헌법 vs 코드 분리**: CLAUDE.md 는 *코드 규정* 만 박제. BLOG_SUPREME_LAW 는 *글 정책*. 책임 경계 명확.
3. **검증 자동화**: 위반 검출이 SQLite `post_analysis` 테이블 런타임 검사로 가능 (제3조 빈 헤더·제4조 이미지 연속 등). `shared/precommit_check.py` 의 `blog` 카테고리 + 게이트 함수 `law_enforcer.enforce_supreme_law()`.
4. **동적 로드**: `law_enforcer.build_writing_rules_block()` 가 매 호출 BLOG_SUPREME_LAW.md 를 *읽어* prompt 상단에 주입. 정책 변경이 *재배포 없이* 즉시 적용.

## 포기한 대안
1. **CLAUDE.md 안에 박기**: 코드 규정과 글 정책이 한 파일에 섞임 → 신규 작업자 혼란. 포기.
2. **작성자별 정책 (jarvis_main.py 안)**: 작성자 N개 → 정책 N벌 → 일관성 불가능. 포기.
3. **dict / YAML 정책**: 정책의 *언어적 뉘앙스* (예: 제0조 감성 도입부 톤) 가 구조화된 형식으로 표현 불가. Markdown 자연어가 최적. 포기.
4. **JSON Schema 검증**: 글의 품질을 *형식 검증* 으로만 잡으면 의미 위반 통과. 결국 LLM 검증 (auditor) 필요. 정책 본문은 LLM 입력용이 더 효율. 포기.
