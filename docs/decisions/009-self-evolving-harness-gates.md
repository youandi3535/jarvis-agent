## 상태

> **v2 변경 (2026-05-17 사용자 박제)**: 초기 박제가 *블로그 발행 흐름* 중심 어휘였음. 사용자 비전 정확 옮김: *모든 동작 공통 — 검증 순환 통과 후만 송출, 송출 후 실패 개념 없음*. Layer 어휘 일반화. Layer 5 "부분 재발행" 제거 — 송출은 종착점.

## 배경

ADR 007 [Self-Evolving Harness 비전](007-self-evolving-harness.md) 의 Stage A (사후 대응 자가 학습) 가 2026-05-16 완료됐다. 그러나 *Stage A 는 오류가 발생한 후* 진단·수정·학습하는 폐쇄 루프. 사용자 비전은 *오류가 외부에 도달하지 않게* 차단하는 단계.

2026-05-17 07:00 KST 경제 브리핑 발행 실패 ([ERRORS 133]) 가 *Stage A 의 한계* 를 드러냈다 — 헌법 14조 게이트 (`enforce_supreme_law`) 가 *작성 후 1회 자동 보정* 만 함. 보정 못한 위반은 *경고만* 보내고 *그대로 발행*. 즉 *직선 시스템*.

사용자 비전을 v2 로 정확 박제:

> *"명령이 들어오면 각자 단계별로 수행하겠지? 그런 다음 결과가 나오겠지? 오류든 아니든... 그 결과가 왔다해서 바로 실행하는 게 아니라, 검증을 통해 어느 단계에서 문제가 있는지 체크해서 순환시키며 문제되는 곳을 수정하고 기록하고, 다시 순환해서 명령의 의도대로 문제없이 정확히 결과가 나오면, 그때 비로써 결과를 내보내는 거야. 송출! 이 시스템이 우리 시스템이잖아."*

> *"모든 동작에 있어 이 검증 순환 시스템이 적용되어야 한다. 결과물은 언제나 항상 의도한대로 나와야 해. 오류가 나오면 안돼. 왜냐면 오류가 발생되면 자체적으로 순환하면서 수정하고 하기 때문이야."*

> *"송출 후 검증? 애초에 실패 자체가 없어. 실패하면 송출 자체가 없을 테니까. 계속 순환하며 오류를 검증하고 잡겠지?"*

## 결정

*Self-Evolving Harness* 를 **Stage B+ (검증 순환 → 송출 패턴)** 로 확장. **모든 동작은 같은 흐름**:

```
[명령 트리거] → [수행 단계 시퀀스] → [1차 결과: 내부]
                                          ↓
                                  ┌─── 검증·수정 순환 ───┐
                                  │  ① 결과 전체 검증    │
                                  │  ② 문제 단계 식별    │
                                  │  ③ GUARDIAN 학습·수정 │
                                  │  ④ 재실행 → ①        │
                                  └────────┬─────────────┘
                                           ↓ (문제 0)
                                       [📤 송출 = 완료]
```

### 핵심 불변 원칙

1. **송출 = 완료 표시**. 외부 도달까지 *포함*된 단일 종착 상태.
2. **결함 있는 결과물은 *영원히 송출되지 않는다***. 검증 순환 안에서만 수정.
3. **송출 후 "실패"라는 개념은 존재하지 않는다**. 외부 응답 실패 = *송출이 아직 일어나지 않은 상태* = 검증 순환의 일부.
4. **모든 명령·모든 트리거·모든 동작에 동일 적용**. 블로그·영상·텔레그램·자유 문장·API — 트리거 무관.

### ★ "하네스" 어휘의 두 층위 — 혼동 방지 (사용자 박제 2026-05-17)

*"Self-Evolving Harness"* 라는 단어가 두 의미로 쓰여 혼동 가능. 명확 박제:

| 층위 | 어휘 | 의미 | 위치 |
|------|------|------|------|
| **광의 (시스템)** | "Self-Evolving Harness 비전" | ADR 007 전체 자가 학습 시스템 — *우산 개념* | ADR 007 + 본 ADR 009 |
| **협의 (파일)** | `harness.py` | Layer 1~4 검증 순환 → 송출 *흐름 엔진* | `JARVIS00_INFRA/harness.py` |

광의 비전 = 협의 인프라 + 학습 두뇌 두 부분품의 *합집합*:

```
ADR 007 비전: "Self-Evolving Harness" (광의)
  │
  ├─ JARVIS00_INFRA/         ← 흐름 *골격* (구조)
  │   ├ preflight.py         — Layer 0 부팅 검증
  │   └ harness.py           — Layer 1~4 검증 순환 → 송출 엔진 (협의)
  │
  └─ JARVIS07_GUARDIAN/      ← 학습 *두뇌* (지능)
      ├ error_collector      — 오류 감지·박제 (catch() 단일 진입점)
      ├ pattern_fixer        — static 6종 + learned_patterns + Contextual Bandit
      ├ error_analyzer       — 2-Tier 분석 (Tier 1 패턴·Bandit → Tier 2 LLM)
      └ ERRORS.md            — 영구 이력
```

### ★ harness.py ↔ GUARDIAN 관계 — 부모/자식 호출

두 부분품은 *수직 분리·수직 협력*:

| 차원 | `JARVIS00/harness.py` | `JARVIS07/GUARDIAN` |
|------|----------------------|--------------------|
| 역할 | 흐름 *골격* | 학습 *두뇌* |
| 질문 | *"어떻게 흐를지"* (단계·검증·재시도·송출) | *"어떻게 수정할지"* (패턴·LLM·학습 누적) |
| 진입 함수 | `run_action(action_def)` | `report(...)` → `apply_fix(error_id)` |
| 데이터 | 동작 state (dict) | `learned_patterns.json` |
| 트리거 | 동작 명령 (모든 트리거 종류) | 오류 발생 (★ harness 의 Layer 3 검증 실패 포함) |

**호출 흐름** (Layer 3 검증 실패 시):
```
harness.run_action() → 검증 실패 감지
    ↓ 호출
error_collector.report(source="harness", context={layer:3, action:..., step:..., kind:..., detail:...})
    ↓
GUARDIAN 의 apply_fix() — 2-Tier 자동 수정 (Tier 1 패턴·Bandit → Tier 2 LLM)
    ↓
learned_patterns 영구 박제
    ↓ 다음 회차
재발 시 LLM 0회 즉시 fix
```

**비유**: harness 는 *자동차의 변속기·브레이크·핸들* (모든 동작이 통과하는 골격) / GUARDIAN 은 *진단 컴퓨터·정비사* (이상 감지·수리·학습) / ADR 007 비전은 *자동차 전체*.

### 다른 작업자 박제 의무
- 새 동작 추가 시: *반드시* `JARVIS00_INFRA/harness.py` 의 `@action_step` + `ActionDefinition` + `run_action()` 사용. *직접* GUARDIAN 호출 금지 — harness 가 *자동으로* 검증 실패 시 GUARDIAN 호출.
- 새 오류 패턴 추가 시: `JARVIS07_GUARDIAN/pattern_fixer.py` 의 `_FIXER_REGISTRY` 갱신. *harness.py 안에 수정 로직 박지 말 것* — 단일 진입점 위반.

### 5 Layer 구조 — 모든 동작 공통

| Layer | 책임 | 구현 위치 |
|-------|------|----------|
| **0** | 시스템 부팅·환경 검증 (도메인 무관, 부팅 1회) | `JARVIS00_INFRA/preflight.py` ✅ 2026-05-17 |
| **1** | 동작 입력·전제조건 (의도 파악·권한·리소스) | `harness.py` `precondition` hook |
| **2** | 수행 단계 실행 (산출물 생성, 동작별 N개 단계) | `harness.py` `@action_step` 시퀀스 |
| **3** | **★ 결과 전체 검증 순환** (문제 0까지 반복) | `harness.py` `verify_loop` (★ 핵심) |
| **4** | 송출 (검증 통과 후만 — 외부 도달까지 포함) | `harness.py` `send()` 콜백 |

★ Layer 5 는 *없음* — 송출이 종착. 외부 응답 실패는 Layer 3 검증 순환 안에서 *송출 미완료* 로 다룬다.

### 동작 종류별 인스턴스 예시

| 동작 | 트리거 | Layer 2 (수행 단계) | Layer 4 (송출) |
|------|--------|--------------------|----------------|
| 블로그 발행 | Cron 07:00 | 데이터→글→이미지→헌법→DB 마커 | 플랫폼 발행 (네이버·티스토리) |
| 영상 제작 | 자유 문장 | 스크립트→렌더→압축→자막 | 유튜브 업로드 |
| 자유 문장 ("대시보드 추가") | 텔레그램 | 의도→코드→테스트→diff | 파일 적용 + 응답 |
| 텔레그램 /status | 명령 | 분류→권한→상태수집→포맷 | 메시지 전송 |
| 로그인 확인 | 사용자 | 쿠키→유효성→실로그인→분석 | 보고서 전송 |

***모두 동일 5 Layer 통과 후 송출***. Layer 2 의 *수행 단계 N개* 가 동작별 다를 뿐, *Layer 3 검증 순환 → Layer 4 송출* 패턴은 *완전 동일*.

## 이유

1. **사용자 비전 정확 박제**: "송출되는 결과는 항상 완벽 — 오류는 순환 안에서 다 해소". *예방 우선* 설계.
2. **모든 동작 일관**: 블로그·영상·텔레그램·자유 문장 등 어떤 트리거든 *같은 게이트 인프라* 사용. 새 동작 추가 시 *수행 단계 N개*만 정의.
3. **GUARDIAN 학습 효과 극대화**: 검증 순환의 *문제 단계 식별 + 수정 패턴* 이 매번 `learned_patterns` 에 누적 → 같은 사고 재발 시 즉시 fix.
4. **7시 사고 type 영구 차단**: 검증 통과 못한 결과물이 외부에 도달할 경로 자체가 *코드 차원에서 봉쇄*.

## 포기한 대안

### 대안 1 — 단순 자동 재시도 (retry decorator)
- *포기 사유*: 오류 원인 *학습 없음*. 같은 원인으로 영원히 재시도. GUARDIAN 진화 가치 0.

### 대안 2 — 단계별 forward gate (Step1 ✅ → Step2 ✅ → ...)
- *포기 사유*: 사용자 비전 정면 반대. 사용자: *"결과 나온 후 전체 검증·순환"* 패턴. 단계 *진행 중* 검증 아님.

### 대안 3 — 송출 후 검증 + 부분 재진입
- *포기 사유*: 사용자 명시 거부 — *"애초에 실패 자체가 없어"*. 외부 응답 실패는 송출 안 됨으로 해석 → 검증 순환의 일부.

### 대안 4 — Stage A 강화만 (사후 대응 똑똑하게)
- *포기 사유*: 사용자 비전 미달. 7시 사고는 *부팅 단계* 폭발이라 사후 대응 불가능 (데몬 자체 못 뜸).
