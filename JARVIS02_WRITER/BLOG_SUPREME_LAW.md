# JARVIS02 블로그 글쓰기 최상위 헌법

> **효력 선언**
> 이 파일의 규정은 JARVIS02가 발행하는 **모든 블로그 글**에 무조건 적용된다.
> `CLAUDE_WRITER.md`, 각 `.py` 파일의 프롬프트, 스케줄 설정 등 **하위 규정과 충돌 시 이 파일이 우선**한다.
> 이 파일은 **수정 금지**. 변경이 필요하면 사용자에게 직접 확인 후 새 버전을 별도 승인받아야 한다.

> **발견 즉시 이 파일로 이관 + 그 위치는 *호출 형태* 로 교체**.

> 허용 호출 형태:
>   1. `law_enforcer.build_writing_rules_block()` — 발행 시 헌법 동적 로드 후 prompt 주입

>   3. `length_manager.*` — spec 위임 alias (TARGET_SENTENCES·MAX_KOREAN·THEME_*)

> 금지: 다른 .py / .md 안에 *분량 숫자 직박제* (예: "30문장", "1500자"), *섹션 이름 박제* (예: "도입부·대장주·면책"),
> *여백 규정 본문* (예: "글↔이미지 1행"), *면책 문구 템플릿*. 모두 본 파일 위임.

## 제0조 — 감성 도입부 (★ 최상위 — 모든 조항보다 우선)

**모든 블로그 글의 시작은 반드시 4문장(약 200자)의 사람 손길이 느껴지는 감성 도입부로 시작한다.**

1. **분량 (★ 사용자 박제 2026-05-15)**: 정확히 **4문장(약 200자)** — 1문장당 약 50자 기준. `length_manager.HUMAN_INTRO_SENTS = 4` + `HUMAN_INTRO_CHARS = 200` 단일 진실 소스.
2. **구조 패턴**: `<p>2문장</p> + [섹션 이미지 ①] + <p>2문장</p>` — 이미지 1개로 호흡 분리. 제4조 교차 배치와 조화.
3. **허용 오프닝**: 개인 일상 관찰, 공감을 부르는 질문, 계절·날씨·분위기 연상, 최근 경험담, 독자에게 건네는 말투.
4. **절대 금지**: 숫자·지표·데이터·퍼센트로 시작, "오늘의 핵심", "이번 주 지표", "최근 경제", "오늘 코스피" 같은 AI식 팩트 오프닝.
5. **예시 (OK)**: "요즘 주변에서 ○○ 얘기가 부쩍 많아졌더라고요. 저도 비슷한 생각이 들어서 자료를 정리해 봤어요. [이미지] 오늘은 그 흐름을 차근차근 짚어보려고 합니다. 함께 들여다보시면 도움이 되실 거예요."
6. **예시 (NG)**: "오늘 코스피는 2,600선에서 출발합니다.", "이번 주 핵심 경제 지표를 정리해 드립니다.", "최근 금리 인상이..."
7. **런타임 집행**: `law_enforcer.py` `check_human_intro()` 가 첫 텍스트 블록 AI식 오프닝 감지 → 텔레그램 경고. 문장 수 미달 시(<4) 추가 폴백 문장 LLM 동적 생성.
8. **프롬프트 의무**: 모든 블로그 작성 프롬프트에 `build_length_phrase(HUMAN_INTRO_SENTS)` = `"4문장(약 200자)"` 박제 (어떤 플랫폼·어떤 글 종류든 예외 없음).

## 제0-B조 — 문단 구조 (★ 최상위 — 제0조와 동급)

**한 문단(단락)에 최대 2문장만 이어 쓸 수 있다.**

1. **2문장 한도**: 한 `<p>` 태그 안에 최대 2문장. 예외 없음.
2. **분리 규칙**: 3문장 → `<p>2문장</p><p>1문장</p>` / 4문장 → `<p>2문장</p><p>2문장</p>` / N문장 → 2문장씩 끊어 각 `<p>`로 분리.
3. **마지막 단락 예외 폐지**: 기존 "마지막 단락만 홀수 허용" 규칙 무효. **모든 단락 2문장 한도 동일 적용**.
4. **적용 범위**: 본문 전체 (도입부·섹션·마무리·면책 모두). 리스트 항목 내 문장도 동일 적용.
5. **프롬프트 의무**: 모든 작성 프롬프트에 이 조항 명시 박제. `<p>(3문장)</p>` 패턴 허용하는 예시 금지.
6. **런타임 집행**: `law_enforcer.enforce_supreme_law()` 내 `_split_overlong_paragraphs()` 가 `<p>` 3문장 이상 감지 → 자동 분리 후 위반 로깅.

## 제1조 — 독창성 (가장 중요)

모든 발행 글은 **독립된 창작물**이어야 한다.

1. **같은 날, 다른 플랫폼(네이버·티스토리)의 글은 서로 달라야 한다.**
   - 제목·소제목·본문 표현이 2곳 모두 다른 관점·다른 어조로 작성.
   - 동일 키워드를 다루더라도 플랫폼별 독자 성격에 맞게 각색.

2. **같은 플랫폼에서 어제 글과 오늘 글의 문장이 같으면 안 된다.**
   - 소제목·본문·CTA·면책 문구 중 어떤 것도 이전 글과 문자 그대로 동일한 문장 사용 금지.

3. **코드에 박힌 고정 한국어 문장은 발행 본문에 들어갈 수 없다.**
   - 모든 발행 문장은 LLM 호출로 생성. 날짜·종목·플랫폼을 프롬프트 시드로 주입하여 매번 다른 결과 보장.
   - 이미지도 동일 원칙 적용 — 고정 템플릿·스타일 풀 사용 금지. 세부 규칙은 제1-B조.
   - 예외: 완전한 LLM 실패 시의 1줄 비상 폴백만 허용 (반드시 LLM 시도 후).

## 제1-B조 — 동적 생성 원칙 (★ 제1조와 동급)

**★ 핵심: 발행 본문의 모든 텍스트와 이미지는 Claude LLM이 매번 동적으로 생성해야 한다. 하드코딩 절대 금지.**

1. **텍스트 동적 생성 의무**: 본문 intro·body·outro·CTA·차트 캡션·이미지 간격 연결 텍스트·면책 문구·감성 도입부 훅 등 외부에 노출되는 모든 한국어 문장은 `shared.llm.invoke_text(alias, prompt)` 호출로만 생성. 고정 문자열 상수·변형 풀 dict 금지.
2. **이미지 동적 생성 의무**: 썸네일·섹션 이미지·인포그래픽·차트 모두 Claude LLM이 배경 프롬프트와 SVG 오버레이를 매번 새로 창작. 고정 템플릿·`STYLE_ZONES`·`THEME_PALETTES` 일체 금지.

## 제2조 — 진실성 (★ 사용자 박제 2026-05-19 강화 → 2026-07-03 수치 중심 재정의)

**★ 원칙 (사용자 박제 2026-07-03 — ADR 013): "숫자로 들어가는 수치 데이터는 무조건
진실되어야 한다. 그것만 조심하면 돼. 글은 상상·추론·예상할 수 있으니 꼭 팩트가
아니어도 된다."** — 하드 게이트 대상은 *수치* 다. 프로즈(서사·전망·해석·상상)는
자유이며, 팩트 데이터를 재료로 매력적·매혹적·감동적으로 쓰는 것이 목표다.

1. **수치·통계는 제공된 데이터에서만 인용** — LLM이 임의로 숫자·백분율·순위·날짜를 창작하는 것은 규정 위반.
   - 허용: "○○지수가 전월 대비 2.3p 상승했다" ← 수집된 실제 데이터에 있는 수치
   - 금지: "약 47.3%의 투자자가…" ← 출처 없는 LLM 창작 수치
2. **추정치는 반드시 "추정" 명시** — 실측치에 "약", "대략", "추정" 등 한정어 붙이지 말 것.
3. **데이터가 없으면 수치를 쓰지 않는다** — 해당 섹션 전체를 제거하거나, 수치 없이 서술형으로 대체.
4. **LLM 생성 문장 vs 실제 데이터 충돌 시 → 데이터 우선**. LLM 문장 수정 또는 삭제.
5. **차트·이미지의 수치도 동일 적용** — 실제 데이터 없이 생성된 차트는 반드시 "※ 참고 이미지 — 실제 수치 아님" 표시.
6. **미완성 표현 발행 금지** — "추가 분석 필요", "데이터 없음", "미정" 같은 표현을 본문에 그대로 남기지 않는다.
7. **★ 비수치 프로즈는 게이트 비대상 (2026-07-03)** — 감상·전망·해석·분석·상상·추론·예상
   등 숫자 없는 서술은 사실성 게이트가 차단하지 않는다. 매력적인 글쓰기의 자유 영역.

**★ 집행** — `law_enforcer.factuality_issues()` (prepublish 게이트) 가 *수치 포함 주장* 만
추출·출처 대조·웹 재검증. 수치 모순·창작 수치만 차단. `audit_factuality()` 는 보조 경고.

## 제4조 — 글과 이미지 배치 규정 (★ 사용자 박제 2026-05-19)

**모든 블로그 글의 문단과 이미지 배치는 아래 4가지 허용 패턴 안에서만 구성한다.**

1. **허용 패턴 1**: `문단 → 이미지 → 문단`
2. **허용 패턴 2**: `문단 → 이미지 → 문단 → 이미지 → 문단`
3. **허용 패턴 3**: `문단 → 문단 → 이미지 → 문단` (문단 2개 연속 후 이미지 — 허용)
4. **허용 패턴 4**: `문단 → 이미지 → 문단 → 문단` (이미지 후 문단 2개 연속 — 허용)

**금지 패턴:**
- **이미지 연속 금지**: `이미지 → 이미지` 어떤 이유로도 발행 불가. 이미지 사이에는 반드시 문단 1개 이상.
- **문단 3개 이상 연속 금지**: 이미지 없이 문단 3개+ 연속 금지. 임계값: `length_manager.MAX_CONSECUTIVE_PARAGRAPHS_WITHOUT_IMAGE = 2`.
- **표도 이미지로 카운트**: 표(`<table>`) 뒤에 즉시 차트/이미지가 오면 이미지 연속. 표 뒤에는 반드시 `<p>` 단락 1개 이상 삽입 후 차트.

**예외**: 소제목 배너 이미지(파일명 `heading_` / `section_title` 포함)는 이미지 카운트 제외.

**런타임 집행**:
- `law_enforcer._dedupe_consecutive_images()` — 이미지 연속 제거.
- `law_enforcer.enforce_image_between_paragraphs()` — 문단 3+ 연속 검출 시 이미지 풀에서 삽입. 풀 부족 시 텔레그램 경고 + 발행 진행.
- `law_enforcer.enforce_supreme_law()` — 위 함수들을 차례로 호출.

**프롬프트 의무**: 모든 작성 프롬프트는 허용 패턴 4가지를 사용. `1:1 교대` 강제 표현 금지.

**섹션별 이미지 최소 보장 (★ 사용자 박제 2026-05-16)**: 섹션 수 ≤ 사용 가능 이미지 수가 되도록 사전 보장. `economic_poster` 의 이미지 생성 함수가 섹션 수 +α 만큼 이미지를 사전 생성.

## 제5조 — 면책 문구 (주의 수칙) 의무

**투자·금융 관련 글의 마지막은 반드시 2문장(약 100자)의 주의 수칙(면책 문구)으로 끝낸다.**

1. **분량 (★ 사용자 박제 2026-05-14)**: 정확히 **2문장(약 100자)** — 1문장당 약 50자 기준. `length_manager.DISCLAIMER_SENTS = 2` + `DISCLAIMER_KOREAN = 100` 단일 진실 소스.
2. **위치**: 글 맨 마지막 (outro 뒤 — 가장 마지막 문단).
3. **내용**: "정보 제공 목적, 투자 권유 아님, 최종 판단과 책임은 본인" 세 가지 의미를 모두 포함.
4. **표현**: 매번·플랫폼별 다양하게 변형 (LLM 자유 생성). 고정 템플릿 절대 금지 (제1-B조 적용).
5. **예시 (OK)**:
   - "본 글은 투자 참고용 정보 제공 목적이며 매수·매도 권유가 아니에요. 모든 투자 결정과 그 결과는 본인의 책임임을 다시 한번 말씀드립니다."
   - "이 글의 내용은 개인적 정리이지 투자 권유가 아닙니다. 실제 매매 판단은 충분히 알아보신 뒤 본인 판단으로 진행해 주세요."

## 제8조 — 분량·섹션·이미지 기준 (★ 사용자 박제 2026-05-17 v2 + 2026-06-01 이미지 최솟값)

**★ 이미지 최솟값 (사용자 박제 2026-06-01 → 2026-07-05 정정 8→5)**:

- 기술 단일 진입점: `JARVIS02_WRITER/post_type_specs.py` 의 `PostTypeSpec.min_images = 5`.
- 코드 상수: `length_manager.MIN_IMAGES = 5` (post_type_specs 파생). 다른 파일 숫자 직박제 금지.

1. **단일 진실 소스**: `JARVIS02_WRITER/post_type_specs.py` 의 `PostTypeSpec` — *추상 범위만* 박제:
   - 글 종류 정의 (purpose·audience·style_hints)
   - 섹션 *수 범위* (min_sections·max_sections)
   - 섹션당 문장 *범위* (sentences_per_section)
   - 필수 섹션 (required_sections — 예: ["면책"])
   - 절대 한계 (max_sentences·min_sentences·max_korean·max_images) — 사용자만 변경
2. **섹션 자체 동적 생성** — `generate_section_plan(spec, topic, context)` 가 매 글마다 LLM 호출로 *주제 맞춤 섹션* 생성:

3. **검증** — LLM 응답이 spec 범위 안인지 자동:
   - 섹션 수 in [min_sections, max_sections]
   - 각 섹션 sentences in sentences_per_section
   - 총 문장 합 in [min_sentences, max_sentences]
   - required_sections 모두 포함
   - 실패 시 max_retries 재요청 → fallback 구조

6. **새 글 종류 추가**: `POST_TYPE_SPECS` dict 에 *purpose 한 줄 + 추상 범위* 박제만. 섹션·분량 모두 자동.
7. **호환 alias**: `length_manager` 의 옛 상수들은 *fallback 평균값* (실제 작성 시 section_plan 우선). 기존 호출자 영향 0.

## 제8-B조 — 분량 표기 표준 (★ 절대 — 사용자 박제 2026-05-14)

**1문장 = 약 50자**. 모든 분량 표기는 *문장 수와 글자 수를 항상 함께* 기록한다. 둘 중 하나만 적는 것 절대 금지.

1. **계산 기준**: `1문장 ≈ 약 50자` (한국어 평균). 상수: `length_manager.KOREAN_PER_SENTENCE = 50`.
2. **표기 패턴**:
   - 단일값: `N문장(약 N×50자)` — 예: `3문장(약 150자)`, `2문장(약 100자)`
   - 범위:   `M~N문장(약 M×50~N×50자)` — 예: `5~6문장(약 250~300자)`, `1~2문장(약 50~100자)`
3. **적용 범위 — 예외 없음**:
   - LLM prompt 안의 분량 지시
   - 코드 주석·docstring 안의 분량 설명
   - CLAUDE.md·BLOG_SUPREME_LAW.md·기타 .md 안의 분량 규정
   - 텔레그램 메시지·로그·DB 메타데이터
4. **금지 표기**:
   - `60자 이내` (문장 수 없음) — 금지
   - `5문장` (글자 수 없음) — 금지
   - `80~120자 한 문장` (구식 표기) — 금지
   - `[가-힣]{N,M}` 정규식·`text[:N]` 본문 자르기 — 제8조 본문에 따라 길이 검사·잘라내기 자체가 금지
5. **단일 진입점**: `JARVIS02_WRITER/length_manager.build_length_phrase(min_sents, max_sents=None)` 헬퍼만 사용.
   - 예: `build_length_phrase(5)` → `"5문장(약 250자)"`
   - 예: `build_length_phrase(1, 2)` → `"1~2문장(약 50~100자)"`
6. **신규 코드 작성 시**: prompt 안 분량 표현은 반드시 `f"{_L.build_length_phrase(N, M)}"` f-string 보간 사용. 하드코딩된 `"N문장"`·`"N자 이내"` 박지 말 것.
7. **검증 명령** (3종 — 모두 결과 0행이어야 함):
   ```bash
   # ① 글자수만 단독 표기 (문장 누락)
   grep -rnE '[0-9]+자\s*(이내|이하|초과|미만|이상)' --include='*.py' JARVIS02_WRITER shared \
     | grep -v length_manager.py | grep -v build_length_phrase | grep -v __pycache__
   # ② 문장만 단독 표기 (글자수 누락) — prompt 내부
   grep -rnE 'f"[^"]*[0-9]+문장[^"]*"' --include='*.py' JARVIS02_WRITER \
     | grep -v '약 [0-9]+자' | grep -v build_length_phrase | grep -v __pycache__
   # ③ 구식 자수 범위 (M~N자)
   grep -rnE '[0-9]+~[0-9]+자' --include='*.py' JARVIS02_WRITER \
     | grep -v length_manager.py | grep -v __pycache__
   ```
8. **위반 시**: prompt 분량 지시가 일관되지 않으면 LLM 출력 분량이 들쭉날쭉 → 발행 품질 저하. 발견 즉시 `build_length_phrase()` 로 교체, 예외 없음.

## 제9조 — 여백 규정 (★ 사용자 박제 2026-05-19)

**★ 핵심 원칙: 문단과 문단, 문단과 이미지, 이미지와 문단 사이는 정확히 1칸(1행) 띄운다. 2칸·3칸 금지.**

   - **글↔글** (문단 → 문단)
   - **글↔이미지** (문단 → 이미지)
   - **이미지↔글** (이미지 → 문단)
   - **이미지↔이미지** (이미지 → 이미지 — 제4조에 의해 연속 자체가 금지이나, 발생 시 동일 적용)
   - **썸네일↔글** (썸네일 다음 첫 문단에도 동일하게 1행)
   - **★ 요약**: "직전이 콘텐츠 블록이고 현재도 콘텐츠 블록이면 무조건 1행 — 2행 이상 절대 금지"
2. **소제목 앞: 예외 — 2행 여백**: `<p>&nbsp;</p><p>&nbsp;</p>` 2줄 삽입. 직전이 문단이든 이미지든 동일.
   - **소제목의 정의**: `<h1>`~`<h6>` HTML 헤더 태그 블록 **또는** 파일명에 `heading_` / `section_` / `economic_h2_` 포함된 소제목 이미지.
3. **블록 내부 과잉 여백 차단**: LLM 또는 플랫폼이 블록 내부에 빈 `<p>` / `<br>` 를 2개 이상 생성하면 1개로 자동 압축.
4. **적용 범위**: 테마글·경제 브리핑 글·트렌드글 전체. 네이버·티스토리 2개 플랫폼 모두.
5. **구현 단일 진입점**: `law_enforcer.enforce_spacing()` — 발행 직전 블록 리스트에 자동 적용.
6. **런타임 집행**: `enforce_supreme_law()` 호출 시 자동 적용. 별도 호출 불필요.
7. **포스터 의무**: 네이버(`post_to_naver` 블록 루프)·티스토리(`post_to_tistory` 블록 루프) 모두 `btype == 'spacer'` 핸들러를 반드시 구현. spacer 블록 무시 = 규정 위반.

## 제12조 — 같은 글 내 시각화 스타일 중복 금지 (★ 절대 — 단일 글 내 모든 이미지/차트/표)

**★ 핵심 원칙: 한 블로그 글(post) 내에서 두 개 이상의 시각화(차트·표·인포그래픽·이미지)가 같은 스타일을 가지면 안 됨.**

1. **적용 범위**:
   - **같은 글의 여러 시각화들** — 표, 바 차트, 라인 차트, 파이 차트, 인포그래픽, 섹션 이미지
   - **색상** (주 데이터 색, 강조색, 배경색)
   - **스타일** (폰트·크기·굵기, 보더·그림자, 레이아웃 패턴)

2. **금지 패턴**:
   ```
   ❌ 같은 글에
   [파란색 + 바 차트] ← 1번
   [파란색 + 라인 차트] ← 2번 (같은 파란색 = 중복!)
   
   ✅ 올바른 예
   [파란색 + 바 차트] ← 1번
   [빨간색 + 라인 차트] ← 2번 (색상 다름)
   [초록색 + 파이 차트] ← 3번 (색상 다름)
   ```

3. **구현 의무** — `jarvis_main.py` / `economic_poster.py`:
   - 글 생성 시작: `used_colors = []` 초기화
   - 각 시각화 생성 시: `_get_dynamic_colors(topic, exclude=used_colors)` 호출
   - 반환된 색상을 `used_colors` 에 추가
   - 발행 직전: `post.visual_metadata = {"colors": used_colors}` DB 저장

4. **색상 추적 메커니즘**:
   ```python
   # 글 생성 시작
   used_colors = []
   
   # 이미지/차트 1 생성
   colors_1 = invoke_text("writer_fast", f"colors for bar chart, avoid {used_colors}")
   used_colors.extend(colors_1)
   
   # 이미지/차트 2 생성 (제외 색상 주입)
   colors_2 = invoke_text("writer_fast", f"colors for line chart, avoid {used_colors}")
   used_colors.extend(colors_2)
   ```

5. **위반 시**: 같은 글의 여러 이미지가 같은 색·같은 스타일 → 독자 "AI 티" 즉시 감지 → 신뢰도 추락 + SEO 저품질. 발견 즉시 이미지 재생성.

## 제13조 — 데이터 표현 일관성 (★ 절대 — 모든 수치·표현)

**숫자·기간·통계 표현은 동일 글 내 일관성 유지. 다른 글과도 충돌 없도록 표준 표현 사용.**

1. **수익률 표현** *(테마주 전용)*: 종목 수익률은 *3개월 수익률만* 표기. 1개월·6개월·1년 수익률 언급 금지 — 데이터 신뢰성·일관성 보장.
2. **재무 수치 본문 기재 금지** *(테마주 전용)*: 현재가·시가총액·PER·ROE·영업이익률·매출액·순이익·부채비율 등 재무 수치 및 재무 관련 설명을 본문 텍스트에 기재 금지. 이미지(차트·표)로 이미 제공되므로 본문 반복 불필요. 정성적 표현('흑자 기조', '고성장', '안정적 수익성')으로 대체.
3. **면책 문구 어조**: 본문 어조는 플랫폼별 다양(제6조)이지만, **면책 1문장은 모든 플랫폼 격식체(~습니다체) 강제**. 면책은 법적·신뢰 영역이므로 친근 표현 부적합.
4. **숫자·기간 표현**: 글 내 동일 지표는 동일 단위로 (KRW vs 원, % vs 퍼센트). 섞어 쓰지 말 것.
5. **금지 표현**: "추정", "약", "대략" 같은 모호한 한정어는 *실제 추정치일 때만* 사용. 실측치에 붙이지 말 것 — 제2조(진실성) 보강.
6. **위반 시**: 동일 글 내 1·3·6개월 수익률 혼재는 독자 혼란 + AI 작성 티 직결. 발행 직전 후처리 검증 의무.

## 제14조 — 태그 특수기호 절대 금지 (★ 사용자 박제 2026-05-15 — 네이버·티스토리만)

**블로그 글 태그(해시태그)는 한글·영문·숫자만 허용. 특수기호·공백·구두점 *전부* 절대 금지.**

1. **적용 대상**: *네이버*·*티스토리* 발행 시 입력하는 태그 리스트.
2. **허용 문자 (유일)**: `0-9`, `A-Z`, `a-z`, `가-힣` 만. 그 외 *모든 문자* 제거.
3. **금지 문자 (예시)**: 공백 ` `, 가운뎃점 `·`, 슬래시 `/`, 하이픈 `-`, 더하기 `+`, 앰퍼샌드 `&`, 괄호 `()[]{}<>`, 따옴표 `"'` , 콜론·세미콜론 `:;`, 물음표·느낌표 `?!`, 별표 `*`, 퍼센트 `%`, `#@$^~|\` 등 *예외 없음*. 한글 자음·모음 단독(ㄱㄴㄷ, ㅏㅑㅓ) 도 금지 (완성형 가-힣 만).
4. **변환 예시**:
   - `GTX(수도권 광역급행철도)` → `GTX수도권광역급행철도`
   - `경제·브리핑`             → `경제브리핑`
   - `AI/머신러닝`             → `AI머신러닝`
   - `5G·반도체`               → `5G반도체`
5. **빈 결과·중복 처리**: sanitize 후 빈 문자열은 *제외*. 정제 후 중복은 *순서 유지* 첫 등장만 유지. 최대 10개로 컷.
6. **기술 단일 진입점**: `shared/seo.py` 의 `sanitize_tag(s)` / `sanitize_tags(list, max_count=10)` 두 함수가 *유일* 한 정제 경로. 다른 파일에 `s.replace('·', '')` 같은 부분 정제 박지 말 것.
7. **호출 의무**: `post_to_tistory(tags=...)` / `post_to_naver(tags=...)` 호출 *직전* `sanitize_tags()` 통과 후 전달. 4 진입점 (`trend_economic_writer.py` 2곳 + `trend_theme_writer.py` 2곳) 모두 적용.
8. **검증 명령**:
   ```bash
   # 특수기호 직접 정제 잔존 (sanitize_tags 외 부분 정제 금지)
   grep -rnE "\.replace\(['\"]·['\"]" --include='*.py' JARVIS02_WRITER | grep -v sanitize_tag
   # → 결과 0행이어야 함
   ```
9. **위반 시**: 특수기호 포함 태그는 네이버·티스토리 해시태그 인식 실패 + 검색 노출 저해 + 자동 발행 사고 (잘못된 태그 매칭). 발견 즉시 `sanitize_tags()` 호출 추가.

## 제16조 — 블로그 원고 생성 의무 프로세스 (★ 절대 — 모든 플랫폼·모든 글 종류)

**어떤 블로그 글을 쓰든 반드시 아래 2단계를 거쳐야 한다. 예외 없음.**

### 1단계 — 원고 생성 직전: 규정 주입 (의무)

`build_writing_rules_block()` (`law_enforcer.py`) 호출 → `supreme_block` 획득 → LLM 프롬프트 **맨 첫 줄** 배치. 이 과정 없이 원고 생성 LLM 호출 금지. Claude Code SDK 1-pass 포함 모든 모델 동일.

```python
from JARVIS02_WRITER.law_enforcer import build_writing_rules_block as _law_blk
supreme_block = _law_blk()          # 매번 파일 읽기 (캐시 금지)
prompt = f"{supreme_block}\n\n..."  # 프롬프트 최상단 배치
```

### 2단계 — 발행 직전: 후처리 집행 (의무)

`post_to_naver / post_to_tistory` 호출 **직전** `enforce_supreme_law()` 실행 필수.

```python
from JARVIS02_WRITER.law_enforcer import enforce_supreme_law, notify_violations
blocks, violations = enforce_supreme_law(blocks, "naver", "경제 브리핑")
notify_violations(violations, "naver", "경제 브리핑")
```
